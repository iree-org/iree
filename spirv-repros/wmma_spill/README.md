# WMMA f16 SPIR-V JIT spilling

This is a standalone repro derived from `bench/matmul_f16.mlir` to demonstrate excessive scratch use in the IR generated by HIP/COMGR.

LLVM version: 23.0.0git (as of 2026-05-12, IREE's bundled `third_party/llvm-project`)  

Files:

- `aot.ll`: final AOT AMDGPU LLVM IR from the ROCm HSACO path.
- `jit.ll`: final AMDGPU LLVM IR recovered from HIP/COMGR after the SPIR-V VMFB is JIT-compiled.
- `spirv_backend_input.ll`: LLVM IR input to the SPIR-V backend, before HIP/COMGR consumes the SPIR-V.
- `matmul_f16.mlir`: source MLIR used only for provenance.

To parse the aot.ll and jit.ll, the helper script can be used to run `llc` on both files and print static instruction counts:

```sh
./run_llc_repro.py
```
Script Output:

```text
LLVM_BIN=/path/to/llvm/bin
instruction   AOT  JIT   delta
------------  ---  ----  -----
instructions  938  1715  +777
buffer_load   24   0     -24
buffer_store  128  0     -128
global_load   0    24    +24
global_store  0    128   +128
scratch       0    248   +248
v_wmma        128  128   0
s_wait        145  300   +155
v_dual        65   113   +48
```

The equivalent commands to compile and collect scratch instruction counts are:

```sh
llc -mtriple=amdgcn-amd-amdhsa -mcpu=gfx1201 -filetype=obj \
  aot.ll -o /tmp/matmul_f16_aot.o

llc -mtriple=amdgcn-amd-amdhsa -mcpu=gfx1201 -filetype=obj \
  jit.ll -o /tmp/matmul_f16_jit.o

llvm-objdump -d /tmp/matmul_f16_aot.o | grep -c 'scratch_'
llvm-objdump -d /tmp/matmul_f16_jit.o | grep -c 'scratch_'
llvm-objdump -d /tmp/matmul_f16_aot.o | grep -c 'v_wmma'
llvm-objdump -d /tmp/matmul_f16_jit.o | grep -c 'v_wmma'
```


`llc -O3` or `-O2` produces the same counts for this repro. `-O0` and `-O1` do not: they introduce additional scratch instructions in both the AOT and JIT objects.

The files were produced from `matmul_f16.mlir` with:

```sh
iree-compile \
  --iree-hal-target-device=hip \
  --iree-rocm-target=gfx1201 \
  --iree-hal-dump-executable-intermediates-to="/tmp/aot" \
  matmul_f16.mlir \
  -o "/tmp/aot.vmfb"

iree-compile \
  --iree-hal-target-device=hip \
  --iree-rocm-target=gfx1201 \
  --iree-rocm-use-spirv \
  --iree-hal-dump-executable-intermediates-to="/tmp/jit" \
  matmul_f16.mlir \
  -o "/tmp/jit.vmfb"

cp "/tmp/aot/module_matmul_2048x2048x2048_dispatch_0_rocm_hsaco_fb.optimized.ll" \
  aot.ll

cp "/tmp/jit/module_matmul_2048x2048x2048_dispatch_0_rocm_spirv_fb.optimized.ll" \
  spirv_backend_input.ll

```

The JIT-side `.ll` comes from HIP/COMGR's saved temporary LLVM bitcode after running the SPIR-V VMFB once:

```sh
AMD_COMGR_SAVE_TEMPS=1 \
  iree-run-module \
  --module="/tmp/jit.vmfb" \
  --device=hip \
  --function=matmul_2048x2048x2048 \
  --input=2048x2048xf16 \
  --input=2048x2048xf16 \
  > "/tmp/run.out"

COMGR_BC=$(
  ls -t /tmp/comgr-*/output/hip_code_object.spv.bc | while read -r bc; do
    strings "$bc" | grep -q matmul_2048x2048x2048 && { echo "$bc"; break; }
  done
)
llvm-dis "$COMGR_BC" -o jit.ll
```

The exact COMGR temp directory name changes between runs. In the run that produced this repro, `COMGR_BC` resolved to `/tmp/comgr-725652/output/hip_code_object.spv.bc`.


`/opt/rocm-7.1.0/llvm/bin/llc` does not consume these textual `.ll` files as-is because the text is LLVM 23 IR syntax and intrinsic spelling/signatures differ from ROCm 7.1. Use a matching LLVM build for the `.ll` repro.

The optional SPIR-V backend input can be lowered with:

```sh
llc -mtriple=spirv64-amd-amdhsa -O3 -filetype=obj \
  spirv_backend_input.ll -o /tmp/matmul_f16.spv
```
